# Root Cause And CAPA Guide

## Problem Statement

Use a factual statement:

- defect or failure observed;
- affected product, process, lot, line, customer, or date range;
- magnitude and trend;
- requirement violated;
- detection point;
- immediate risk.

Avoid vague statements such as "quality issue" or "operator error".

## Root Cause Discipline

- A root cause must explain why the issue happened and why it escaped if escape occurred.
- Use 5 Why for chain logic, but verify each important branch with evidence.
- Use Ishikawa to expand possible causes, not to prove them.
- Use data stratification to narrow cause by line, shift, supplier, tool, cavity, product, or time.
- Distinguish root cause, contributing cause, detection failure, and systemic cause.

## CAPA Quality Checks

- Corrective action should remove or control the verified cause.
- Preventive action should address recurrence in similar processes when applicable.
- Actions need owner, due date, evidence, document impact, training impact, and verification method.
- Effectiveness must have a metric and review window.
- Do not close CAPA only because the action was implemented; verify the effect.

## Effectiveness Examples

- Defect rate below target for three consecutive lots.
- No recurrence in the next defined production window.
- SPC chart stable after process change.
- Audit finding corrected with evidence of updated procedure and trained users.
- Before-after test shows meaningful improvement and no adverse side effect.
